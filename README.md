# EVS Remote — управление ПК в сети

Мобильное приложение (Flutter) к вашему домашнему парку машин: Wake-on-LAN,
мониторинг состояния через агент, голосовой ввод в EVS. Цель по платформам —
**Android 7.0+ (API 24)** и **iOS**.

В архиве лежит только то, что писалось вручную: весь `lib/`, `pubspec.yaml`,
тесты и готовые нативные конфиги. Платформенные папки `android/` и `ios/`
генерирует сам Flutter командой `flutter create` — так надёжнее, чем возить
их в архиве (Gradle-обёртка и Xcode-проект привязаны к версии SDK).

## Что нужно

- Flutter SDK 3.19+ (`flutter --version`)
- VS Code с расширениями **Flutter** и **Dart**
- Android: Android Studio или командные `cmdline-tools` + эмулятор/устройство
- iOS: только на macOS с Xcode

## Быстрый старт

```bash
# 1. Распакуйте архив и войдите в папку
cd evs_remote

# 2. Сгенерируйте платформенные папки вокруг существующего кода.
#    Замените com.yourname на свой домен в обратной записи.
flutter create --platforms=android,ios --org com.yourname .

# 3. Установите зависимости
flutter pub get
```

Команда `flutter create .` создаёт `android/` и `ios/`, но **перезапишет**
`lib/main.dart` шаблонным счётчиком. Восстановите наш `lib/` и `pubspec.yaml`
из архива поверх сгенерированных (просто скопируйте с заменой). После этого
`lib/main.dart` снова наш.

> Порядок надёжнее наоборот: сначала `flutter create` в пустой папке, затем
> скопировать в неё `lib/`, `pubspec.yaml`, `test/`, `analysis_options.yaml`
> из архива с заменой.

## Нативная настройка (обязательно)

Приложение ходит к агенту по **http** (LAN и Tailscale), а не https — обе
платформы по умолчанию это блокируют. Плюс нужен доступ к сети и локальной сети.

### Android — `minSdk 24` и разрешения

Откройте `android/app/build.gradle` (или `build.gradle.kts`) и задайте minSdk:

```groovy
// Groovy DSL
android {
    defaultConfig {
        minSdkVersion 24
    }
}
```
```kotlin
// Kotlin DSL (build.gradle.kts) — в новых версиях Flutter
android {
    defaultConfig {
        minSdk = 24
    }
}
```

Затем замените `android/app/src/main/AndroidManifest.xml` на готовый из
`native/android/AndroidManifest.xml`. В нём уже прописаны разрешение INTERNET,
`usesCleartextTraffic="true"`, RECORD_AUDIO для будущего голоса и
POST_NOTIFICATIONS для push-алертов.

### Android — Firebase (push-уведомления об алертах)

Push для Android идёт через self-hosted **ntfy** на вашем сервере, настроенный
на релей через ваш собственный бесплатный Firebase-проект (без этого шага
приложение соберётся и заработает как обычно — просто без push, алерты
по-прежнему видны в приложении на переднем плане):

1. Создайте проект в [Firebase Console](https://console.firebase.google.com),
   добавьте в него Android-приложение с `applicationId`, который вы указали в
   `flutter create --org ...` (`<org>.evs_remote`).
2. Скачайте `google-services.json` и положите его в `android/app/google-services.json`
   (после `flutter create`, так как `android/` не хранится в репозитории).
3. Подключите Gradle-плагин Google services — добавьте classpath/plugin в
   `android/build.gradle(.kts)` и `android/app/build.gradle(.kts)`. Точный
   синтаксис зависит от версии Flutter Gradle Plugin — см. текущую
   документацию [FlutterFire](https://firebase.flutter.dev) и
   [Firebase для Android](https://firebase.google.com/docs/android/setup).
4. На сервере настройте self-hosted ntfy на релей push через этот же
   Firebase-проект (ключи конфигурации ntfy меняются между версиями — см.
   актуальную документацию [ntfy self-hosting](https://docs.ntfy.sh/install/)).

### iOS — App Transport Security и локальная сеть

Откройте `ios/Runner/Info.plist` и добавьте внутрь корневого `<dict>` ключи из
`native/ios/Info-additions.plist` (ATS, доступ к локальной сети, микрофон).
Минимальную версию iOS поднимите до **15.0** (требует Firebase).

> Если собираете не локально, а в облаке (для AltStore) — эти же ключи и версия
> проставляются автоматически, вручную ничего делать не нужно (см. «Сборка IPA
> в облаке» ниже).

Если под рукой нет Android SDK, APK собирается в CI — workflow
`.github/workflows/build-apk.yml` (запуск вручную через вкладку **Actions →
Build APK → Run workflow**, либо автоматически при пуше кода в `main`). Готовый
`app-release.apk` кладётся в артефакты запуска (подписан debug-ключом —
устанавливается сайдлоадом).

Для Android папка `android/` **закоммичена и настроена** (в отличие от `ios/`,
которая по-прежнему генерируется `flutter create`): applicationId
`com.kekw2077.evs_remote`, `minSdk 24`, core library desugaring для
`flutter_local_notifications`, Gradle-плагин Google services для FCM.

Push (Firebase) требует `google-services.json` — он **не хранится в репозитории**
(в `.gitignore`), а подставляется в CI из секрета репозитория
`GOOGLE_SERVICES_JSON`. Подготовка:

1. Создайте бесплатный проект в [Firebase Console](https://console.firebase.google.com),
   добавьте Android-приложение с package name **`com.kekw2077.evs_remote`**,
   скачайте `google-services.json`.
2. Закодируйте в base64 и положите в секрет репозитория
   (**Settings → Secrets and variables → Actions → New repository secret**,
   имя `GOOGLE_SERVICES_JSON`):
   ```bash
   base64 -w0 google-services.json   # Linux; на macOS: base64 google-services.json
   ```
3. Запустите workflow. Без секрета сборка остановится с понятной ошибкой.

## Сборка IPA в облаке (iOS / AltStore)

iOS-версия собирается в CI на macOS-раннере — workflow
`.github/workflows/build-ios.yml` (запуск вручную через **Actions → Build iOS →
Run workflow** либо автоматически при пуше в `main`). На выходе —
**неподписанный `nexus-unsigned.ipa`** в артефактах запуска.

Ставится сайдлоадом через [AltStore](https://altstore.io) или SideStore: они
переподписывают приложение вашим Apple ID прямо на устройстве, поэтому платный
**Apple Developer Program не нужен**. Бесплатная подпись живёт 7 дней —
AltStore обновляет её автоматически, пока рядом есть настроенный AltServer.

Как и с Android, папка `ios/` **не хранится в репозитории** — workflow
генерирует её на лету (`flutter create`) и патчит:

- bundle id `com.kekw2077.evs_remote`, имя приложения **Nexus**;
- `Info.plist` — ATS (агент по `http://`), доступ к локальной сети (WoL + агенты),
  разрешение на микрофон — из `native/ios/Info-additions.plist`;
- минимальная версия **iOS 15.0** (требование Firebase);
- иконка приложения из `assets/icon/` (см. ниже).

**Ограничения бесплатной подписи (важно):**

- **Push-уведомления не работают** — APNs недоступен без платного аккаунта.
  Код это учитывает (`PushService` на iOS сразу выходит), сборка не ломается.
  Алерты на iOS принимайте через официальное приложение
  [ntfy](https://ntfy.sh) (подписка на ваш топик) — см. «Push-уведомления».
- Приложение нужно **переустанавливать/переподписывать раз в 7 дней** (AltStore
  делает это сам при активном AltServer).

**Иконка под iOS.** iOS сам скругляет углы (маска-суперэллипс) и запрещает
прозрачность, поэтому отдельный «скруглённый» PNG не нужен — наоборот, нужен
полноразмерный квадрат. `flutter_launcher_icons` с `remove_alpha_ios` заливает
прозрачные углы мастера тёмным фоном контейнера (`background_color_ios`); его
радиус совпадает с маской iOS, так что заливка попадает в обрезаемую зону и на
готовой иконке не видна. Генерируется автоматически в шаге сборки.

Локальная сборка APK на машине с Android SDK работает так же, но требует
положить `google-services.json` в `android/app/` вручную.

## Обновление приложения (встроенный апдейтер)

Источник обновлений — **GitHub Releases** этого репозитория. В приложении:
**Настройки → Обновления → «Проверить обновления»**.

- Приложение запрашивает последний релиз (`/releases/latest`), сравнивает его
  версию с текущей (semver) и, если релиз новее, показывает версию и список
  изменений.
- **Android:** кнопка «Скачать и установить» качает APK-ассет релиза и открывает
  системный установщик (нужно разрешение **«Установка неизвестных приложений»**
  для Nexus — система попросит один раз; в манифесте есть
  `REQUEST_INSTALL_PACKAGES`).
- **iOS:** установка APK невозможна — показывается ссылка на страницу релиза,
  обновление ставится через AltStore.

Чтобы проверка версий работала корректно, **тег релиза = версия приложения**:
`release.yml` собирает APK с `--build-name` из тега (без префикса `v`), поэтому
внутренняя версия APK всегда совпадает с релизом. При выпуске новой версии
просто укажите тег в **Actions → Release → Run workflow** (например `v0.1.1`).

> Первый APK (v0.1.0) был собран до этого правила и внутри помечен как `1.0.0` —
> такой билд не увидит `0.1.x` как более новую версию. Достаточно один раз
> вручную поставить следующий релиз `0.1.x` (с апдейтером внутри), дальше
> обновления пойдут сами.

## Как это связано с агентом

Приложение ждёт на каждой отслеживаемой машине HTTP-агент:

```
GET  /health              -> {status,hostname} (без токена)     — доступность, определение загрузки, имя хоста
GET  /metrics             -> {cpu,ram,disk,disks:[{name,percent,totalBytes,usedBytes}],cpuTemp?,gpuTemp?,...} — Bearer-токен
GET  /alerts?device=ID    -> {alerts:[{id,level,message}]} — по порогам устройства + Nextcloud, Bearer-токен
GET  /alert-config?device=ID -> {cpu,ram,disk,temperature,localOverride} — эффективные пороги устройства, Bearer-токен
PUT  /alert-config        <- {deviceId,topic?,scope,cpu?,ram?,disk?,temperature?} — Bearer-токен
GET  /nextcloud           -> {configured,reachable,maintenance,version,...} — состояние Nextcloud, Bearer-токен
POST /wake                -> {mac,broadcast,port}       — ретрансляция magic-пакета
```

Пороги алертов — **multi-tenant**: у агента есть общий default + оверрайды по
устройствам (`deviceId`). `PUT /alert-config` со `scope`:
`all` — общие пороги (для всех), `device` — оверрайд только для этого телефона,
`clear` — снять оверрайд, `register` — просто зарегистрировать топик устройства
(чтобы получать push). Телефон генерит стабильный `deviceId` и свой топик
(`nexus-<id>`) один раз и хранит в prefs; топик пер-девайс, вручную не задаётся.

Это тот самый `agent slim.py` в корне репозитория. Токен из
`/etc/pc-agent.env` вводится в карточке компьютера (вкладка «Статус») и в
настройках ретранслятора. Пороги алертов настраиваются в карточке компьютера —
телефон источник истины, агент сохраняет присланное (`pc-agent-config.json`
рядом с `agent slim.py`, путь через `PC_AGENT_STATE_FILE`). При сохранении
изменённых порогов приложение спрашивает: применить **для всех устройств**
(общий default) или **только для этого телефона** (оверрайд по `deviceId`).

### Агент на основном ПК (Windows) — `agent_pc.py`

`agent slim.py` рассчитан на Linux (`/proc`, `statvfs`, `uname`). Для основного
ПК на **Windows 11** в корне лежит `agent_pc.py` — тот же HTTP-контракт
(health / metrics / alerts / alert-config / wake), но метрики читаются через
WinAPI (`ctypes`), а температура/загрузка/VRAM видеокарты — через `nvidia-smi`
(идёт с драйвером NVIDIA). Тоже **только стандартная библиотека**, ни pip, ни
venv не нужны. Nextcloud-часть тут не запускается (её нет на ПК).

Запуск в PowerShell:

```powershell
$env:PC_AGENT_TOKEN="ваш-токен"       # тот же токен вводится в карточке ПК
python agent_pc.py                     # или pythonw agent_pc.py — без окна консоли
```

Автозапуск — через «Планировщик заданий» (триггер «При входе в систему»,
действие `pythonw agent_pc.py`) или ярлык в `shell:startup`. Переменные
(`PC_AGENT_PORT`, `PC_AGENT_DISK`, `PC_AGENT_ALERT_*`, `PC_AGENT_NTFY_URL` и
т.д.) — как у Linux-агента. Открыть порт в брандмауэре Windows:

```powershell
New-NetFirewallRule -DisplayName "PC Agent" -Direction Inbound -Protocol TCP -LocalPort 8765 -Action Allow
```

Особенности Windows: отдаются **все фиксированные диски** (C:, D:, …) в `disks[]`;
температура **GPU** (`gpuTemp`) — через `nvidia-smi`; температура **CPU**
(`cpuTemp`) — best-effort через ACPI-датчик (WMI `MSAcpi_ThermalZoneTemperature`),
на многих платах недоступна/врёт, тогда просто не отдаётся (точная температура
CPU требует LibreHardwareMonitor и т.п., которые не тянем). Загрузка GPU
(`gpu`, %) и видеопамять (`vramUsedBytes`/`vramTotalBytes`) тоже отдаются —
приложение рисует их отдельными полосками. `loadAvg` пустой — в Windows его нет.

**Метрики всех дисков и температур (обе платформы).** `/metrics` отдаёт массив
`disks[]` со всеми физическими томами (Linux — реальные `/dev`-устройства из
`/proc/mounts` без псевдо-ФС и loop; Windows — все фиксированные диски), а также
`cpuTemp`/`gpuTemp` по отдельности. Приложение показывает каждый диск отдельной
полоской и обе температуры; алерт по диску срабатывает для **любого** тома выше
порога. Поля `disk`/`temperature` сохранены для обратной совместимости со старыми
версиями приложения.

### Push-уведомления об алертах (опционально), per-device

Если на сервере задать `PC_AGENT_NTFY_URL` (адрес self-hosted ntfy), агент раз
в `PC_AGENT_WATCH_INTERVAL` секунд (по умолчанию 10) идёт по всем
зарегистрированным устройствам и шлёт **каждому в его топик** его алерты — по
его эффективным порогам (общий default или личный оверрайд). Телефон
регистрируется автоматически (свой `deviceId` + топик `nexus-<id>`) при первом
опросе хоста; отдельно топик вводить не нужно. Так «только для этого телефона»
работает и для push: сервер шлёт этому устройству по его порогам, другим — по
общим. См. «Android — Firebase» выше для брендированной интеграции push.
**iOS**: без Apple Developer Program push в приложении не работает — установите
официальное приложение ntfy и подпишитесь на топик своего устройства
(`https://<ваш-ntfy-домен>/nexus-<id>`); экран «Состояние» по-прежнему
показывает алерты, пока приложение открыто.

### Мониторинг Nextcloud (опционально)

Если Nextcloud крутится на той же машине, что и агент, задайте на сервере
`PC_AGENT_NC_URL` (адрес облака, например `https://cloud.example.com`) — агент
раз в `PC_AGENT_NC_INTERVAL` секунд (по умолчанию 60) читает `/status.php`
(доступность, режим обслуживания, версия) и, если задан `PC_AGENT_NC_TOKEN`
(токен приложения **serverinfo**), подробную статистику через serverinfo API
(пользователи, активные за 5м/1ч/24ч, файлы, общие ресурсы, свободное место,
обновления приложений, версии PHP/БД/веб-сервера, размер БД). Отдельная вкладка
**«Облако»** показывает полный дашборд, а компактный блок — в карточке
компьютера. Проблемы (облако недоступно, режим обслуживания, требуется апгрейд
БД, обновления приложений/ядра, предупреждения настройки) добавляются в общий
поток алертов — то есть тоже попадают в push через ntfy. **Учётные данные
Nextcloud остаются на сервере** (в env-переменных агента), в телефон не передаются.

Токен serverinfo генерируется на сервере Nextcloud:
```
occ config:app:set serverinfo token --value "$(openssl rand -hex 32)"
```
(в Nextcloud AIO — через `docker exec ... occ ...`). Без токена работает только
`/status.php` — доступность, режим обслуживания и версия.

**Глубокие проверки (обновление ядра, предупреждения настройки/безопасности)**
требуют доступа агента к `occ` — задайте `PC_AGENT_NC_OCC` с префиксом вызова:
```
# Nextcloud AIO:
PC_AGENT_NC_OCC="docker exec --user www-data nextcloud-aio-nextcloud php occ"
# обычный Nextcloud:
PC_AGENT_NC_OCC="sudo -u www-data php /var/www/nextcloud/occ"
```
Агент вызовет `occ update:check` (обновление ядра) и `occ setupchecks`
(предупреждения). Для варианта с `docker exec` агенту нужен доступ к Docker
(состоять в группе `docker` или запускаться от root) — это фактически права
root, учитывайте при настройке. Без `PC_AGENT_NC_OCC` эти проверки просто не
запускаются.

### Пути Wake-on-LAN

- **Прямой (LAN)** — телефон сам шлёт magic-пакет по UDP. Работает, когда
  телефон в той же сети (дома по Wi-Fi). В поле адреса — broadcast подсети
  (например `192.168.1.255`), порт по умолчанию 9. На приёмной стороне порт
  не важен: magic-пакет ловит сетевая карта на аппаратном уровне, ОС спит.
- **Через ретранслятор** — телефон просит всегда-включённый сервер
  (`POST /wake`) отправить пакет в LAN. Нужен из внешней сети, где
  широковещание не проходит. Настройки → «Ретранслятор Wake-on-LAN». Не
  обязательно через Tailscale — включите «HTTPS (публичный адрес)». Будит
  **любую машину, кроме самого сервера-ретранслятора**.
- **Из интернета через роутер (WAN)** — для машины, которую релей не
  разбудит (в первую очередь сам сервер, когда он выключен). В поле адреса
  укажите **DDNS/публичный адрес**, в поле порта — **проброшенный на роутере
  порт**, и включите тумблер **«Напрямую, минуя ретранслятор»**. Приложение
  резолвит имя и шлёт magic-пакет прямо на этот адрес. Тумблер важен: без него
  при включённом релее пакет ушёл бы через релей (тот бродкастит в свою LAN, а
  не на ваш публичный адрес). Флаг задаётся у каждой цели отдельно, так что
  LAN- и WAN-машины уживаются в одном списке.

**Настройка роутера для WAN-пробуждения:** нужен статический ARP-маппинг для
машины (чтобы роутер знал, куда слать пакет, пока она спит и не отвечает на
ARP) + проброс порта `WAN:порт → IP_машины:9`. Затем в приложении вводите свой
DDNS/публичный IP и этот порт. Оговорки: глобальный broadcast
`255.255.255.255` через WAN не проходит (нужен проброс на конкретный IP); если
провайдер за CGNAT (серый IP), проброс невозможен — остаётся только релей через
Tailscale. Часть роутеров (Keenetic, Mikrotik, OpenWrt, ASUS-Merlin) умеют слать
WoL сами по расписанию/SSH — это альтернатива пробросу.

### Поиск машин в сети

На вкладке «Состояние» — кнопка **поиска** (иконка радара / «Найти в сети»).
Сканирует вашу Wi-Fi-подсеть на отклик агента (`GET /health`) и показывает
найденные машины с кнопкой «Добавить» — она открывает форму мониторинга с уже
подставленными адресом и портом (остаётся ввести токен). Имя берётся из
`hostname` в ответе `/health` (нужен свежий агент; старый отдаёт только статус —
тогда показывается IP).

> Находка — **локальный** IP (работает дома). Чтобы отслеживать машину из любой
> точки, после добавления замените адрес на её **Tailscale-имя** (`*.ts.net`)
> или другой адрес, доступный снаружи. С мобильной песочницы MAC узнать нельзя,
> поэтому поиск наполняет мониторинг, а не список WoL.

## Структура

```
lib/
  core/        токены темы, форматирование, валидация, генератор id
  models/      WolTarget, MonitoredHost, HostMetrics, AlertItem, AlertConfig, NextcloudStatus
  services/    prefs_store, agent_client (http), wol_sender (udp), push_service (FCM+local-notifications), device_identity, network_scanner (поиск агентов в LAN)
  state/       четыре ChangeNotifier-контроллера (provider)

native/        готовые AndroidManifest.xml и Info.plist-ключи
```

## Что уже реально, что пока условно

- **EVS** (`evs_controller.dart`) — открывает настоящий WebSocket
  (`ws://host:port/mobile`, пакет `web_socket_channel`). Формат сообщений
  (`{"type": "command"|"recognized", "text": ...}`) провизорный — финализируется
  вместе с функцией приёма на стороне десктопного EVS, которой пока нет.
- **Метрики и алерты** — реальный `fetch` для `/metrics`/`/alerts`, пороги
  настраиваются per-host и multi-tenant (общие / только для устройства) и реально
  пушатся на агент (`PUT /alert-config`).
- **Push-уведомления** (`push_service.dart`) — Android: Firebase + `flutter_local_notifications`,
  показывает алерт даже когда приложение свёрнуто/закрыто, тап открывает
  вкладку «Состояние». iOS: не реализовано в приложении (см. «Push-уведомления
  об алертах» выше) — осознанный компромисс ради экономии $99/год Apple
  Developer Program, а не недоработка.
- **Ретранслятор WoL** — доступен по HTTPS без Tailscale, если включить
  «HTTPS (публичный адрес)» в настройках и обеспечить внешнюю доступность
  сервера (обратный прокси + проброс порта).
- **Визуализатор голоса** (`waveform.dart`) — амплитуда это RMS из PCM16-потока
  микрофона (пакет `record`), не синтетика.
- **Ничего из push/алертов/HTTPS-ретранслятора не проверено на реальном
  устройстве** — машина разработки без Android SDK/Visual Studio, `agent slim.py`
  не запустить на Windows. Проверено только `flutter analyze`/`flutter test`/
  `python -m py_compile`.
