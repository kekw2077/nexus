# Nexus — развёртывание серверной части (runbook для чата с SSH-доступом к серверу)

> **Как пользоваться:** открой этот файл в новом чате с Claude, где подключён
> SSH MCP к домашнему Ubuntu-серверу, и скажи: «разверни всё по этому runbook,
> сначала проверь свободные порты и предложи их — я проброшу их на роутере».
> Документ самодостаточен: в нём весь контекст, ничего не нужно объяснять заново.

---

## 0. Контекст (прочитай первым)

**Проект:** Nexus (он же EVS Remote) — мобильное Flutter-приложение для
Wake-on-LAN и мониторинга домашних машин. Репозиторий:
`https://github.com/kekw2077/nexus` (недавно переименован из `evs_remote`).
Клиентская часть уже написана и закоммичена — **на сервере ничего из Dart
разворачивать не нужно**, только серверные компоненты ниже.

**Что уже умеет приложение (для понимания, зачем это всё):**
- Опрашивает на каждой машине HTTP-агент (`agent slim.py`) — метрики и алерты.
- Будит машины: напрямую UDP-броадкастом в своей Wi-Fi сети, а из внешней сети —
  через **ретранслятор** (агент на всегда включённой машине шлёт magic-пакет по её LAN).
- Показывает push-уведомления об алертах через self-hosted **ntfy** + собственный
  **Firebase**-проект (брендированный push на Android). iOS-интеграции нет
  сознательно (без Apple Developer Program) — на iOS ставится официальное
  приложение ntfy.
- Пороги алертов и топик ntfy настраиваются в приложении **per-host** и пушатся
  на агент — телефон источник истины.
- **Мониторинг Nextcloud:** агент на сервере опрашивает `status.php` + serverinfo
  API облака и отдаёт состояние в приложение; проблемы (недоступно, обслуживание,
  апгрейд БД, обновления) идут в общий поток алертов и push. Учётные данные
  Nextcloud — только в env агента, не в телефоне. См. §4.5.

**Что разворачиваем на сервере (3 компонента):**
1. **`agent slim.py`** — как systemd-сервис на всегда включённом Ubuntu-сервере.
   Этот же агент служит ретранслятором WoL для остальных машин в LAN.
2. **ntfy** — Docker-контейнер, self-hosted push-сервер, релеит в FCM через
   Firebase-проект пользователя.
3. **Обратный прокси с TLS** (Caddy) — терминирует HTTPS для публичного адреса
   ретранслятора (WoL из внешней сети) и для ntfy.

**Инфраструктура (важные факты и ограничения):**
- У пользователя **статический публичный IP** и **свой домен**, который уже
  используется для развёрнутого на этом же сервере **Nextcloud AIO**.
- На сервере ~32 Docker-контейнера. **Nextcloud AIO почти наверняка занимает
  порты 80/443** своим встроенным прокси. **Не сломай работающий Nextcloud** —
  это критично. Не переконфигурируй его reverse-proxy без явной необходимости и
  без бэкапа.
- Пользователь сам добавит A-записи в свой домен и сам пробросит порты на роутере
  — **но только после того, как ты проверишь, какие порты на сервере свободны,
  и предложишь конкретные**.

**Поток данных (для проверки правильности настройки):**
```
WoL из внешней сети:
  телефон ──HTTPS──> [роутер:проброшенный_порт] ──> Caddy ──HTTP loopback──> agent:8765 ──UDP broadcast──> спящий ПК

Метрики/алерты в LAN:
  телефон ──HTTP──> agent:8765 (напрямую по локальному IP, порт НЕ пробрасывается наружу)

Push-алерт:
  agent (watcher-поток) ──POST──> ntfy ──> FCM (Firebase пользователя) ──> Android-приложение Nexus
```

## 1. Pre-flight: сначала обследуй сервер, ничего не предполагай

Выполни и покажи пользователю результаты, прежде чем что-либо ставить:

- ОС и версия: `lsb_release -a`, `uname -a`
- Docker: `docker ps --format '{{.Names}}\t{{.Ports}}'` — увидеть, что запущено и
  какие порты уже заняты контейнерами (особенно найти Nextcloud AIO и понять,
  держит ли он 80/443).
- Занятые порты на хосте: `sudo ss -tulpn | sort -t: -k2 -n` — составить список
  занятых TCP-портов.
- Кто слушает 80/443: `sudo ss -tulpn | grep -E ':80 |:443 '`
- Есть ли уже reverse-proxy (Caddy/Traefik/nginx-proxy-manager) в стеке:
  `docker ps | grep -iE 'caddy|traefik|nginx|proxy'` — если да, **возможно его и
  надо переиспользовать вместо второго прокси** (реши по факту, см. §4).
- Файрвол: `sudo ufw status verbose` (или проверь iptables/nftables).
- Есть ли Python 3: `python3 --version` (нужен ≥3.9 для агента — он на stdlib,
  ни pip, ни venv не требует).
- Сетевые данные для WoL: `ip -br addr` и `ip route` — определить LAN-подсеть и
  broadcast-адрес (понадобятся для проверки пробуждения).

**Итог этого шага:** список свободных портов-кандидатов + понимание, занимает ли
Nextcloud AIO 80/443 и есть ли уже общий reverse-proxy.

## 2. Решения, которые нужно вынести пользователю (до установки)

Предложи пользователю (через вопрос), он подтвердит / поправит:

1. **Два сабдомена** его существующего домена, например:
   - `relay.<домен>` — публичный HTTPS-адрес ретранслятора WoL
   - `ntfy.<домен>` — self-hosted ntfy
   (A-записи на статический IP пользователь добавит сам — дай ему точные имена.)
2. **Публичные порты для проброса на роутере.** Если Nextcloud AIO держит 80/443
   — **не отбирай их**. Рекомендация по умолчанию: выделить прокси Nexus
   **отдельные порты** (например 8443 для HTTPS-входа) и пробросить именно их —
   приложение это поддерживает, порт ретранслятора настраивается в его настройках,
   ничего не завязано на 443. Предложи конкретные свободные порты из §1.
3. **Firebase service account JSON** для ntfy — нужен файл ключа из
   Firebase-проекта пользователя (тот самый проект, что он заводит для push).
   Попроси пользователя сгенерировать его (Firebase Console → Project settings →
   Service accounts → Generate new private key) и передать на сервер. Без него
   ntfy не сможет релеить push в FCM. **Не выдумывай содержимое — дождись файла.**

> ⚠️ **Порядок с TLS:** Let's Encrypt (которым пользуется Caddy) не выдаёт
> сертификаты на голый IP — нужен домен, указывающий на IP, и открытый путь для
> ACME-проверки. Если для проброса выбраны нестандартные порты (не 80/443), для
> Caddy используй **DNS-01 challenge** (через API DNS-провайдера пользователя)
> либо, если провайдер держит 80 занятым Nextcloud'ом, согласуй ACME-механизм с
> пользователем. Это уточняется по факту — не полагайся на HTTP-01 вслепую.

## 3. Компонент 1 — агент как systemd-сервис

Агент **должен работать на хосте напрямую (systemd), не в контейнере**:
- ему нужен UDP-броадкаст в LAN для WoL (в контейнере — только с `network_mode: host`);
- он читает `/proc`, `/sys/class/hwmon`, `os.getloadavg()` — в контейнере это были
  бы метрики контейнера, а не хоста.

### 3.1 Файлы
```bash
sudo mkdir -p /opt/pc-agent /var/lib/pc-agent
# скопировать "agent slim.py" из репозитория на сервер как /opt/pc-agent/agent.py
# (имя файла в репозитории — с пробелом: "agent slim.py")
```

### 3.2 Токен и env
```bash
# сгенерировать токен (запиши его — он же вводится в приложении в карточке машины):
python3 -c 'import secrets; print(secrets.token_urlsafe(24))'
```

Создай `/etc/pc-agent.env` (права 600, владелец root или сервис-юзер):
```ini
PC_AGENT_TOKEN=<сгенерированный_токен>
PC_AGENT_HOST=0.0.0.0
PC_AGENT_PORT=8765            # проверь, что свободен; наружу НЕ пробрасывается
PC_AGENT_DISK=/
PC_AGENT_WAKE=1
# ВАЖНО: абсолютный путь в каталог, куда сервис-юзер может писать.
# По умолчанию агент пишет относительный "pc-agent-config.json" в CWD —
# под systemd это сломается. Задай явно:
PC_AGENT_STATE_FILE=/var/lib/pc-agent/config.json
# push (заполнишь после развёртывания ntfy, см. §4):
PC_AGENT_NTFY_URL=https://ntfy.<домен>
PC_AGENT_WATCH_INTERVAL=10
# мониторинг Nextcloud (агент и Nextcloud на одном сервере, см. §4.5):
PC_AGENT_NC_URL=https://cloud.<домен>       # адрес облака (можно внутренний)
PC_AGENT_NC_TOKEN=<токен_serverinfo>        # см. §4.5; без него — только status.php
PC_AGENT_NC_INTERVAL=60
# occ-проверки (обновление ядра + предупреждения безопасности), см. §4.5:
PC_AGENT_NC_OCC=docker exec --user www-data nextcloud-aio-nextcloud php occ
# seed-пороги (необязательно — телефон всё равно перезапишет через /alert-config):
# PC_AGENT_ALERT_CPU=90
# PC_AGENT_ALERT_RAM=90
# PC_AGENT_ALERT_DISK=90
# PC_AGENT_ALERT_TEMP=85
```

### 3.3 systemd-unit
`/etc/systemd/system/pc-agent.service`:
```ini
[Unit]
Description=Nexus PC Agent (metrics + WoL relay + ntfy watcher)
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
EnvironmentFile=/etc/pc-agent.env
ExecStart=/usr/bin/python3 /opt/pc-agent/agent.py
Restart=on-failure
RestartSec=3
# сервис-юзер должен иметь право писать в PC_AGENT_STATE_FILE:
StateDirectory=pc-agent          # создаёт/чинит права на /var/lib/pc-agent
DynamicUser=yes
# WoL требует broadcast — DynamicUser это позволяет; если возникнут проблемы с
# доступом к /sys/class/hwmon (температура), можно убрать DynamicUser и указать
# User=root (температура тогда будет читаться; это неприватные данные).

[Install]
WantedBy=multi-user.target
```

> ⚠️ **Готча с температурой vs DynamicUser:** под `DynamicUser` чтение
> `/sys/class/hwmon` обычно работает, но если температура не отдаётся —
> попробуй без `DynamicUser` (тогда пропиши `User=` с правами на
> `/var/lib/pc-agent`). Проверь по факту через `curl /metrics`.

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now pc-agent
systemctl status pc-agent --no-pager
```

### 3.4 Проверка агента (локально, без наружного доступа)
```bash
TOKEN=<токен>
curl -s localhost:8765/health                                   # {"status":"ok"}
curl -s -H "Authorization: Bearer $TOKEN" localhost:8765/metrics | python3 -m json.tool
curl -s -H "Authorization: Bearer $TOKEN" localhost:8765/alert-config
# задать пороги/топик (имитация телефона):
curl -s -X PUT -H "Authorization: Bearer $TOKEN" -H 'Content-Type: application/json' \
  -d '{"cpu":90,"ram":90,"disk":90,"temperature":85,"ntfyTopic":"nexus-server"}' \
  localhost:8765/alert-config
```

## 4. Компонент 2 — ntfy (Docker) + Firebase relay

Цель: агент публикует push в ntfy → ntfy релеит в FCM через Firebase-проект
пользователя → уведомление приходит в приложение Nexus на Android.

### 4.1 Развёртывание контейнера
Ставь `binwiederhier/ntfy` через Docker (compose-файл в стиле существующего
стека). Ключевые моменты конфигурации `server.yml`:
- `base-url: https://ntfy.<домен>` — публичный адрес.
- **Firebase relay:** ntfy умеет пересылать сообщения в FCM. Нужен путь к
  service account JSON пользователя (§2.3).

> ⚠️ **Не фиксируй названия ключей конфига ntfy по памяти.** Точные имена
> (например, ключ для файла Firebase service account) менялись между версиями
> ntfy. **Сверься с актуальной документацией ntfy по self-hosting + Firebase**
> непосредственно при настройке: раздел про `firebase-key-file` / FCM. Загрузи
> текущую доку (`https://docs.ntfy.sh/config/`) и возьми имена ключей оттуда, а
> не отсюда.

Смонтируй service account JSON внутрь контейнера (read-only) и укажи путь к нему
в конфиге. Данные ntfy (подписки/кэш) — на volume.

### 4.2 Прописать ntfy в агенте
После того как ntfy доступен, впиши его адрес в `/etc/pc-agent.env`
(`PC_AGENT_NTFY_URL=https://ntfy.<домен>`) и перезапусти агент:
`sudo systemctl restart pc-agent`. Watcher-поток стартует только если задан
`PC_AGENT_NTFY_URL` **и** через `/alert-config` установлен непустой `ntfyTopic`.

### 4.3 Проверка ntfy
```bash
# базовая публикация (должна дойти до подписчиков топика):
curl -s -d "тест из runbook" https://ntfy.<домен>/nexus-server
```
Полную цепочку до телефона (FCM → приложение) можно проверить только с реальным
Android-устройством с установленным Nexus и настроенным Firebase.

### 4.5 Мониторинг Nextcloud (агент уже на том же сервере)

Nextcloud крутится на этом же сервере, поэтому агент может опрашивать его сам.
Учётные данные остаются на сервере, в телефон не уходят.

- `PC_AGENT_NC_URL` — адрес облака. Можно указать внутренний адрес (loopback/имя
  контейнера), чтобы не гонять трафик через публичный домен; главное, чтобы
  агент по нему достучался. Проверь: `curl -s <NC_URL>/status.php`.
- **Токен serverinfo** (для подробной статистики — пользователи, файлы, свободное
  место, обновления). Сгенерируй на стороне Nextcloud:
  ```bash
  # обычный Nextcloud:
  sudo -u www-data php occ config:app:set serverinfo token --value "$(openssl rand -hex 32)"
  # Nextcloud AIO (найди имя контейнера nextcloud-aio-nextcloud):
  docker exec -it nextcloud-aio-nextcloud php occ config:app:set serverinfo token --value "$(openssl rand -hex 32)"
  ```
  Полученный токен впиши в `PC_AGENT_NC_TOKEN` и перезапусти агент
  (`sudo systemctl restart pc-agent`). Без токена работает только `status.php`
  (доступность/обслуживание/версия) — тоже полезно, но без статистики.

> ⚠️ **Структура ответа serverinfo немного меняется между версиями Nextcloud.**
> Агент читает её защищённо (отсутствующие поля → просто не показываются). Проверь
> реальный ответ на своей версии:
> `curl -s -H 'OCS-APIRequest: true' '<NC_URL>/ocs/v2.php/apps/serverinfo/api/v1/info?format=json&token=<токен>' | python3 -m json.tool`
> — и, если ключи (`nextcloud.storage.num_users`, `activeUsers.last5minutes`,
> `nextcloud.system.apps.num_updates_available`, `server.php.version`,
> `server.database.type/version/size` и т.п.) отличаются, поправь разбор в
> `fetch_nextcloud()` в `agent slim.py` под факт.

**occ-проверки (обновление ядра + предупреждения настройки/безопасности).**
Требуют, чтобы агент мог выполнить `occ`. Для Nextcloud AIO это `docker exec` в
контейнер `nextcloud-aio-nextcloud`, поэтому:
- **агенту нужен доступ к Docker** — добавь его сервис-юзера в группу `docker`
  **или** запускай агент от root. ⚠️ Членство в группе `docker` фактически даёт
  права root на хосте — осознанное решение; если это неприемлемо, оставь
  `PC_AGENT_NC_OCC` пустым (occ-проверки просто не запустятся, остальное работает).
  Под `DynamicUser=yes` в systemd (см. §3.3) доступа к docker-сокету не будет —
  для occ придётся убрать `DynamicUser` и задать конкретного юзера в группе
  `docker`, либо `User=root`.
- проверь, что префикс работает и команды существуют на твоей версии NC:
  ```bash
  docker exec --user www-data nextcloud-aio-nextcloud php occ update:check
  docker exec --user www-data nextcloud-aio-nextcloud php occ setupchecks --output=json | python3 -m json.tool
  ```
  `setupchecks` появился в Nextcloud 28+. Если его нет или JSON другой формы —
  агент просто не выставит поле предупреждений (не падает); при желании поправь
  разбор в `_occ_checks()` в `agent slim.py`.

Проверка эндпоинта агента:
```bash
curl -s -H "Authorization: Bearer $TOKEN" localhost:8765/nextcloud | python3 -m json.tool
# ожидается configured:true, reachable:true, version:..., при токене hasServerinfo:true,
# при PC_AGENT_NC_OCC — поля coreUpdateAvailable/warningsCount/warnings
```

## 5. Компонент 3 — обратный прокси с TLS (Caddy)

Даёт наружу два HTTPS-адреса: `relay.<домен>` → агент, `ntfy.<домен>` → ntfy.

### 5.1 Ключевое решение по портам (не сломать Nextcloud AIO)
- **Если 80/443 заняты Nextcloud AIO** (наиболее вероятно): подними **отдельный
  Caddy** на нестандартном публичном порту (например 8443) и проброшенном на
  роутере, с DNS-01 для сертификатов. Приложение поддерживает произвольный порт
  ретранслятора — это чистое, неинвазивное решение. **Не трогай прокси Nextcloud.**
- **Если в стеке уже есть общий reverse-proxy** (Traefik / nginx-proxy-manager /
  свой Caddy), который держит 80/443 и через который ходит Nextcloud — **добавь
  два vhost/route туда**, вместо второго прокси. Это предпочтительнее, если такой
  прокси есть и в него безопасно добавить маршруты.
- Определись по результатам §1 и **согласуй с пользователем**, прежде чем менять
  что-либо, связанное с Nextcloud.

### 5.2 Пример Caddyfile (отдельный Caddy на своих портах)
```caddy
relay.<домен> {
    reverse_proxy 127.0.0.1:8765     # агент (WoL-ретранслятор)
}
ntfy.<домен> {
    reverse_proxy 127.0.0.1:<порт_ntfy>
}
```
(Слушающие порты Caddy и ACME-механизм — под выбранную в §2 схему портов;
для нестандартных портов настрой DNS-01.)

### 5.3 Файрвол
Открой на хосте только выбранные публичные порты (например 8443), плюс то, что
уже было. `PC_AGENT_PORT` (8765) наружу **не** открывай — к нему ходит только
Caddy по loopback и телефон по LAN.

## 6. Что делает ПОЛЬЗОВАТЕЛЬ (не Claude)

Собери в конце для пользователя единый список:
1. **DNS:** добавить A-записи `relay.<домен>` и `ntfy.<домен>` → статический IP.
2. **Роутер:** пробросить выбранные публичные порты (например 8443) на внутренний
   IP сервера. `PC_AGENT_PORT` **не** пробрасывать.
3. **Firebase:** передать на сервер service account JSON (§2.3), если ещё не.
4. **В приложении Nexus:**
   - В карточке сервера: адрес (LAN-IP для дома) + `PC_AGENT_PORT` + токен из §3.2,
     задать пороги алертов и `ntfyTopic` (тот же, что проверял в §3.4).
   - В настройках → «Ретранслятор Wake-on-LAN»: включить, адрес `relay.<домен>`,
     публичный порт (например 8443), тот же токен, **включить тумблер HTTPS**.

## 8. Известные неопределённости — свериться с доками при реализации

- **Ключи конфига ntfy для Firebase relay** — брать из актуальной
  `https://docs.ntfy.sh/config/`, не по памяти (§4.1).
- **Взаимодействие с Nextcloud AIO** — как именно он занимает 80/443 и есть ли у
  него режим работы за внешним reverse-proxy; выяснить на месте, не ломать (§5.1).
- **ACME-механизм** при нестандартных портах — вероятно DNS-01; зависит от
  DNS-провайдера пользователя (§2).
- **Температура под DynamicUser** — проверить `curl /metrics`, при отсутствии
  `temperature` убрать DynamicUser (§3.3).

## 9. Правила безопасности

- **Не трогай работающий Nextcloud AIO** без бэкапа и явного согласия пользователя.
  При любом изменении общего reverse-proxy — сначала бэкап конфига.
- Токен агента — секрет; `/etc/pc-agent.env` права 600; не логируй токен.
- Наружу пробрасывается только HTTPS-порт Caddy, не сырой порт агента.
- Service account JSON Firebase — секрет; монтируй read-only, права ограничь.
